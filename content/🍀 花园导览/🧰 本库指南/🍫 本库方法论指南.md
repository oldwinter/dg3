# 🍫 本库方法论指南

本库当前默认方法论不是 ACCESS、PARA、CODE、Daily Notes 或 LLM Wiki 中的任何一个，而是一条更适合 AI agents 接手的操作链路：

```text
Capture -> Route -> Compile -> Canonicalize -> Execute -> Review
```

旧方法论仍然有价值，但它们现在是局部工具和历史参考，不再互相争夺全库默认组织法。

## 默认链路

| 环节 | 放哪里 | 负责什么 | 不负责什么 |
|---|---|---|---|
| Capture | `Calendar/Daily notes/`、当天临时记录 | 当天流水、现场上下文、临时想法 | 长期百科、账号配置、大段数据样例 |
| Route | [[这条笔记该去哪]]、`Cards/`、`Sources/`、`Spaces/`、`Atlas/`、`Calendar/Tasks/` | 判断内容应该进入哪一层 | 把所有碎片都塞进 Cards 或 Daily |
| Compile | [[Atlas/LLM Wiki/_index]]、[[Atlas/LLM Wiki/_playbook]] | 来源摘要、问题、跨来源综合 | 全库分类、任务状态、私人流水 |
| Canonicalize | `Atlas/MOCs/`、`Spaces/2-Area/`、`🍀 花园导览/` | 稳定定义、长期结构、公开入口 | 保留临时过程或低信号摘抄 |
| Execute | `Calendar/Tasks/`、[[Atlas/Bases/项目推进工作台.base]] | 任务状态、日期、项目归属、推进证据 | 项目背景长文和学习材料 |
| Review | `Calendar/Plan & Review/`、`_system/docs/plans/`、`_system/memory/` | 周期复盘、计划、过程记忆 | 成为长期知识正本 |

## 人和 agent 怎么共用

给 AI agent 用时，本库需要入口少、规则清楚、文件边界稳定；给人自己用时，本库仍然要允许长时间阅读、深度思考和专题学习。

所以规则不是“所有笔记都要短”，而是：

- 原文、书籍批注、外部剪藏可以长，但要保留在 `Sources/` 或明确的来源层。
- 计划流水、AI summary、日志样例、JSON 样例不应该长期卡在一个几千行 Markdown 里；应该拆成当前摘要、历史归档、附件或来源索引。
- 深度思考可以先写长，但成熟以后要沉淀成 1 个入口定义、2 到 4 条核心判断和少量高价值双链。
- 任务状态不要藏在长文里；可执行事项进入 `Calendar/Tasks/`，由 TaskNotes / Base 视图管理。

## 旧方法论的当前位置

| 旧方法 | 现在怎么用 |
|---|---|
| 卡片笔记 / 常青笔记 | 适合稳定概念和短判断，不适合把所有 Daily stray 都变成 Card |
| PARA | 仍可理解 `Projects / Areas / Resources / Archive` 顶层语义，但不直接决定每条笔记的去处 |
| CODE | 作为输入到输出的启发；Capture 仍有用，Distill / Express 已拆给 LLM Wiki、MOC 和发布层 |
| ACCESS | 保留为早期 Obsidian 组织经验，不再作为全库默认导航法 |
| Daily Notes 工作流 | 降级为 Capture 和时间线，不再承载长期正本 |
| LLM Wiki | 只做 Compile 层：source summary、query、synthesis 和 reading map |
